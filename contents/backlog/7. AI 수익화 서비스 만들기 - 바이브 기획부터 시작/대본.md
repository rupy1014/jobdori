# AI 수익화 서비스 만들기 - 바이브 기획부터 시작

## [0:00-0:15] HOOK - 콘솔 명령어의 한계

"콘텐츠 공장 만들었는데 DB도 없고 API도 없습니다.

진짜 자동화는 앱으로 만들 때 가능합니다."

## [0:15-0:45] PROBLEM - 파일 시스템의 3가지 한계

지난 5편까지 우리가 만든 것:

```
/factory-run → 콘텐츠 100개 자동 생산
→ 파일로 저장 (raw-data/, produced-content/...)
```

근데 이 방식의 문제:

**문제 1: 데이터 관리 불가능**
- 파일 1,000개 쌓이면? → 찾기도 힘듦
- "지난달 생성한 재테크 콘텐츠" 검색? → 불가능
- 통계? → 파일 일일이 세어야 함

**문제 2: API 연동 어려움**
- 문자 알림 보내려면? → 별도 스크립트 작성
- 블로그 자동 발행? → 파일 읽고 → API 호출 → 복잡
- 실시간 모니터링? → 파일 열어봐야 함

**문제 3: 확장 불가능**
- 여러 프로젝트 동시 관리? → 폴더 헷갈림
- 설정 변경? → 파일 찾아서 수동 수정
- 과거 기록 추적? → 파일 이름으로만 추측

**핵심 문제:**

파일 시스템 = **단순 저장**만 가능
DB + API = **검색, 연동, 자동화** 가능

## [0:45-1:30] AGITATE - 진짜 자동화는 따로 있다

**지금 우리 시스템:**
```
콘솔 명령어 → 파일 생성 → 사람이 확인
```

**진짜 자동화 시스템:**
```
앱 → DB 저장 → API 연동 → 완전 무인 운영
```

**구체적 차이:**

현재 (파일 기반):
```
1. /factory-run 실행
2. produced-content/ 폴더 열기
3. 파일 하나씩 확인
4. 블로그에 복사 붙여넣기
5. 문자는... 직접 폰으로 메모
```

이상적 (DB + API):
```
1. 앱에서 "실행" 버튼 클릭
2. DB에 자동 저장
3. 블로그 API로 자동 발행
4. 문자 API로 결과 알림
5. 대시보드에서 실시간 확인
```

**핵심 인식전환:**

파일 = 임시 저장소
DB = 영구 데이터베이스 + 검색 + 통계

API = 외부 서비스 자동 연동

**이걸 배우려면 앱을 만들어봐야 합니다.**

## [2:00-3:30] VALUE DELIVERY 1 - DB가 뭔가?

### 비개발자를 위한 쉬운 설명

**파일 시스템 vs 데이터베이스**

**파일 시스템 (지금까지 방식):**
```
produced-content/
├── 2025-11-01-article-1.md
├── 2025-11-01-article-2.md
├── 2025-11-02-article-3.md
└── ... (1,000개)
```

**할 수 있는 것:**
- 파일 저장
- 파일 읽기
- 폴더로 분류

**할 수 없는 것:**
- "재테크 주제만" 검색 ❌
- "합격률 95% 이상만" 필터링 ❌
- "지난 달 생산량" 통계 ❌

**데이터베이스 (DB):**
```sql
contents 테이블
┌────┬───────────┬────────┬──────┬─────────┐
│ ID │ 제목       │ 주제   │ 품질 │ 생성일   │
├────┼───────────┼────────┼──────┼─────────┤
│ 1  │청년적금... │재테크  │ 95%  │11-01    │
│ 2  │전세vs월세 │부동산  │ 92%  │11-01    │
│ 3  │ETF 추천   │주식    │ 98%  │11-02    │
└────┴───────────┴────────┴──────┴─────────┘
```

**할 수 있는 것:**
- 검색: "재테크 주제만" ✅
- 필터링: "품질 95% 이상만" ✅
- 통계: "지난 달 생산량" ✅
- 정렬: "품질 높은 순" ✅

### DB가 필요한 이유

**실전 시나리오:**

파일 시스템:
```
Q: "재테크 주제로 지난 달 몇 개 만들었지?"
A: 폴더 열어서 → 파일 이름 보고 → 일일이 세기
   → 30분 소요
```

DB:
```sql
SELECT COUNT(*)
FROM contents
WHERE 주제='재테크'
AND 생성일 BETWEEN '2025-10-01' AND '2025-10-31'

→ 결과: 23개 (1초)
```

### DB 핵심 개념 3가지

**1. 테이블 (Table)** = 엑셀 시트
```
contents 테이블 = 콘텐츠 정보 저장
users 테이블 = 사용자 정보 저장
projects 테이블 = 프로젝트 정보 저장
```

**2. 컬럼 (Column)** = 엑셀 열
```
ID | 제목 | 주제 | 품질 | 생성일
```

**3. 로우 (Row)** = 엑셀 행
```
1 | 청년적금 | 재테크 | 95% | 11-01
```

**⚠️ 3분 리텐션 훅:**

"DB는 엑셀이라고 생각하세요. 근데 검색이 미친듯이 빠른 엑셀..."

## [3:30-5:00] VALUE DELIVERY 2 - API가 뭔가?

### 비개발자를 위한 쉬운 설명

**API = 프로그램 간 소통 방법**

**비유:**

레스토랑에서:
```
손님 (나) → 웨이터 (API) → 주방 (서비스)
         주문             요리
손님 ← 웨이터 ← 주방
      음식 전달
```

프로그램에서:
```
내 앱 → API → 블로그 서비스
      "글 발행해줘"
내 앱 ← API ← 블로그
      "발행 완료"
```

### API가 필요한 이유

**파일 시스템 (수동):**
```
1. 콘텐츠 생성 완료
2. produced-content/article.md 열기
3. 내용 복사
4. 블로그 접속
5. 붙여넣기
6. 발행 버튼 클릭
```
→ 매번 수작업

**API 연동 (자동):**
```python
# 코드 한 줄
blog_api.publish(title, content)
```
→ 자동 발행

### 실전 API 예시

**1. 블로그 발행 API**
```
내 앱 → 티스토리 API
       { "제목": "청년 적금", "내용": "..." }

티스토리 → "발행 완료, URL: ..."
```

**2. 문자 알림 API**
```
내 앱 → 알리고 SMS API
       { "번호": "010-1234-5678", "내용": "콘텐츠 50개 완료" }

알리고 → "전송 완료"
```

**3. Claude AI API**
```
내 앱 → Claude API
       { "프롬프트": "재테크 글 써줘" }

Claude → { "결과": "2024년 청년 적금..." }
```

### API 핵심 개념

**1. 요청 (Request)** = 주문
```
"이거 해줘"
```

**2. 응답 (Response)** = 결과
```
"했어요, 결과는 이거"
```

**3. 엔드포인트 (Endpoint)** = 메뉴판
```
/api/publish    → 글 발행
/api/delete     → 글 삭제
/api/stats      → 통계 조회
```

**💡 5분 리텐션 훅:**

"DB로 저장하고, API로 연동하면, 완전 자동화가 가능합니다..."

## [5:00-6:30] VALUE DELIVERY 3 - 아키텍처 설계

### 시스템 구조 설계

**전체 아키텍처:**

```
┌──────────────┐
│  프론트엔드   │  ← 사용자가 보는 화면 (앱)
│  (Vibe)      │
└──────┬───────┘
       │ API 호출
┌──────▼───────┐
│  백엔드       │  ← 비즈니스 로직 (자동화)
│  (Claude)    │
└──────┬───────┘
       │
   ┌───┴────┬─────────┬──────────┐
   ▼        ▼         ▼          ▼
┌─────┐ ┌─────┐  ┌──────┐  ┌──────┐
│ DB  │ │Claude│  │블로그 │  │문자  │
│     │ │ API │  │ API  │  │ API  │
└─────┘ └─────┘  └──────┘  └──────┘
```

### 용어 설명

**프론트엔드 (Frontend)**
- 사용자가 보는 화면
- 버튼, 입력창, 디자인
- 예: 웹사이트, 앱 화면

**백엔드 (Backend)**
- 눈에 안 보이는 뒷단
- 실제 작업 수행 (콘텐츠 생성, DB 저장)
- 예: 서버, 자동화 스크립트

**DB (Database)**
- 데이터 저장소
- 콘텐츠, 사용자 정보 저장

**API**
- 프론트엔드 ↔ 백엔드 소통
- 백엔드 ↔ 외부 서비스 소통

### 콘텐츠 공장 아키텍처

**데이터 흐름:**

```
1. 사용자 → 프론트엔드: "생성" 버튼 클릭

2. 프론트엔드 → 백엔드 API:
   POST /api/generate
   { "주제": "재테크", "개수": 5 }

3. 백엔드:
   a. Playwright로 데이터 수집
   b. Claude API로 콘텐츠 생성
   c. 검증 Agent 실행
   d. DB에 저장

4. 백엔드 → 프론트엔드 API:
   { "status": "완료", "결과": [...] }

5. 프론트엔드 → 사용자:
   "5개 생성 완료!" 알림 + 결과 화면

6. (선택) 백엔드 → 문자 API:
   "콘텐츠 5개 완료" 문자 전송
```

### 기획 산출물 만들기

**개발 전 필수 문서:**

**1. 요구사항 정의서**
```markdown
# 콘텐츠 공장 앱 요구사항

## 기능
- F1: 사용자가 주제를 입력하면 콘텐츠 자동 생성
- F2: 생성된 콘텐츠를 DB에 저장
- F3: 생성 이력 조회 (검색, 필터링)
- F4: 블로그 자동 발행 (API 연동)
- F5: 문자 알림 (API 연동)

## 제약사항
- 응답 시간: 5초 이내
- 동시 사용자: 10명
- 데이터 보관: 1년
```

**2. DB 스키마 설계**
```sql
-- contents 테이블
CREATE TABLE contents (
  id INT PRIMARY KEY,
  title VARCHAR(200),
  topic VARCHAR(50),
  content TEXT,
  quality_score INT,
  status VARCHAR(20),  -- 'draft', 'approved', 'published'
  created_at DATETIME
);

-- projects 테이블
CREATE TABLE projects (
  id INT PRIMARY KEY,
  name VARCHAR(100),
  tone VARCHAR(50),    -- 톤앤매너
  created_at DATETIME
);
```

**3. API 명세서**
```yaml
# /api/generate (콘텐츠 생성)
POST /api/generate
Request:
  {
    "topic": "재테크",
    "count": 5,
    "tone": "친근함"
  }
Response:
  {
    "status": "success",
    "contents": [
      { "id": 1, "title": "...", "content": "..." },
      ...
    ]
  }

# /api/contents (목록 조회)
GET /api/contents?topic=재테크&limit=10

# /api/publish (블로그 발행)
POST /api/publish
```

## [8:00-9:00] SOCIAL PROOF - 학습 효과

### DB/API를 왜 앱으로 배우는가?

**기존 학습 방법:**

**1. 책/강의로 배우기:**
```
DB 이론 → SQL 문법 → API 개념 → REST 원리
→ 이해는 됨
→ 근데 실전에서 못 씀
→ "어디에 어떻게 써야 하지?"
```

**2. 바이브 앱으로 배우기:**
```
"콘텐츠를 DB에 저장하고 싶다"
→ 테이블 설계 (contents, projects)
→ 바이브로 CRUD 구현
→ 실제 데이터 저장/조회 성공

"블로그에 자동 발행하고 싶다"
→ API 연동 설계 (/api/publish)
→ 바이브로 API 호출 구현
→ 실제 발행 성공
```

### 실전 학습의 위력

**이론만 배운 사람:**
- DB = 테이블, SQL
- API = 요청/응답
- 근데 실전에서 막막

**앱 만들면서 배운 사람:**
- "콘텐츠 검색하려면 DB 테이블이 필요하구나"
- "블로그 발행하려면 API 연동이 필요하구나"
- "아, 그래서 아키텍처 설계를 먼저 하는구나"

**차이:**
- 이론: "DB가 뭐다" (암기)
- 실전: "DB를 이렇게 쓴다" (활용)

### 바이브로 배우는 3가지 장점

**1. 실전 중심**
- 책: "SQL 문법은..." (이론)
- 바이브: "콘텐츠 저장하려면 이 테이블 필요" (목적)

**2. 빠른 피드백**
- 전통 개발: 코드 짜고 → 실행 → 에러 → 수정 (30분)
- 바이브: 화면 그리고 → 바로 확인 → 수정 (3분)

**3. 개념 이해**
- 코딩만: "이게 왜 안 되지?" 삽질
- 바이브 기획: "아, DB/API가 이래서 필요하구나" 깨달음
