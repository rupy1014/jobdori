### 바이브 코딩, 뭐가 다른가

개발자들이 왜 그렇게 빠른지 보세요.

옛날 개발:
- 설계 → 코딩 → 테스트 → 수정
- 완벽하게 설계하고, 완벽하게 만들려고 함

바이브 코딩:
- AI한테 "이거 만들어줘" → 결과 보고 → "이 부분 고쳐줘" → 반복
- **일단 만들고, 고치고, 또 고치고**

핵심이 뭐냐면요,

**완벽하게 설계한 다음에 만드는 게 아니에요.**
**만들면서 설계가 구체화되는 거예요.**

### "AI랑 대화하기"는 뻔한 얘기

*(핵심)*

- 예쁜 표, 다이어그램

바이브 시대 산출물:
- 마크다운 문서
- 구조화된 텍스트
- **AI가 이해하기 좋은 형태**

왜냐면요,

근데 이제는요?

개발자가 기획서를 AI한테 주면서 "이거 만들어줘" 해요.

그러면 AI가 이해하기 좋은 형태로 만들어야죠.

> **마크다운으로, 구조화된 텍스트로, AI가 바로 읽을 수 있게.**

이게 바이브 기획의 핵심이에요.

| 전반부 | 후반부 |

| **핵심** - 새로운 방식 | **현실** - 회사가 요구하는 것 |

## [Part 2] 마크다운으로 기획하기 - 10:00 - 18:00

### Claude Code: 터미널에서 일하는 AI

제가 쓰는 도구부터 소개할게요.
**Claude Code**라는 겁니다.

일반적인 AI 서비스랑 뭐가 다르냐면요.

ChatGPT나 웹 Claude는
**브라우저에서 채팅**하잖아요.
대화하고, 결과 복사하고, 내 파일에 붙여넣고.

Claude Code는 다릅니다.
**터미널에서 바로 작업**해요.

내 프로젝트 폴더에서 대화하면서
**파일을 직접 읽고, 직접 만들어줍니다.**

복사-붙여넣기가 없어요.

> 왜 이게 중요하냐면,
> **맥락이 끊기지 않아요.**

웹 AI는 매번 새 채팅 열면 처음부터 다시 설명해야 하잖아요.
Claude Code는 프로젝트 전체를 알고 있어요.
이전에 만든 파일, 폴더 구조, 다 기억합니다.

---

### 지침 파일: AI한테 업무 매뉴얼 주기

그런데 맥락을 기억한다고 끝이 아니에요.
**어떻게 일하라고 알려줘야** 합니다.

Claude Code에는 **지침 파일**이라는 게 있어요.

**CLAUDE.md** - 프로젝트 전체 지침
- "우리 서비스는 이런 거야"
- "타겟 고객은 이런 사람들이야"
- "문서는 이런 형식으로 써줘"

**rules/** 폴더 - 세부 지침들
- 기능정의서 작성 규칙
- 페르소나 분석 템플릿
- 세일즈 메시지 포맷

> 신입한테 업무 매뉴얼 주는 거랑 똑같아요.
> **한 번 잘 만들어두면, 매번 설명 안 해도 됩니다.**

---

### 루프백: 점점 똑똑해지는 지침

여기서 중요한 게 있어요.
**지침은 한 번에 완성되지 않아요.**

개발자들이 코드 짜는 방식 있잖아요.
일단 짜고, 에러 나면 고치고, 다시 돌려보고.

지침도 똑같이 해요.

```
1. 일단 지침 써서 시켜본다
2. 결과 보고 아쉬운 점 찾는다
3. 지침에 추가한다
4. 새 채팅 열어서 다시 시켜본다
5. 반복
```

이걸 **루프백**이라고 불러요.

> 대화하면서 발견한 좋은 패턴을
> **지침으로 저장하고,**
> **새 채팅에서 다시 테스트하고,**
> **계속 개선하는 것.**

처음엔 지침이 별로여도 괜찮아요.
3번만 루프백 돌리면 확 달라집니다.

### Custom Command: 나만의 단축키

그런데 Claude Code의 진짜 힘은
**Custom Command**에 있어요.

한국말로 하면 **"나만의 단축키"**입니다.
 

> **매번 AI한테 같은 방식으로 물어보는 게 있다면,**
> **원하는 산출물의 퀄리티나 양식대로 뽑길 원한다면,**
> **그걸 명령어 하나로 만들어두는 것.**

### 예시: 제가 만든 명령어들

저는 기획할 때 항상 이렇게 물어봐요:

- "서비스 컨셉 정리해줘"
- "타겟 고객 분석해줘"
- "핵심 문제랑 해결책 정리해줘"

매번 이걸 다 치기 귀찮잖아요?

그래서 하나의 명령어로 만들었어요.

```
/appkit.new 야놀자 같은 테니스 코트 예약 앱
```

이 한 줄이면:
- 서비스 컨셉 정리
- 타겟 고객 분석
- 핵심 문제/해결책 도출
- **마크다운 문서로 자동 생성**

*(직접 시연)*

## [Part 3] 전통적 산출물로 변환하기 - 18:00 - 24:00

### 현실: 회사는 PPT를 원한다

자, 마크다운으로 기획했어요.
AI가 읽기 좋은 형태로 만들었어요.

근데 상사한테 보고할 때 마크다운 보여줄 순 없잖아요.

> "이게 뭐야? PPT로 만들어와."

이런 말 들을 거예요.

그래서 **변환**이 필요합니다.

### 마크다운 → 전통 산출물

좋은 소식은요,

**마크다운에서 전통 산출물로 바꾸는 건 쉬워요.**

왜냐면 내용은 이미 다 있거든요.
형태만 바꾸면 돼요.

```
"이 마크다운 문서를 PPT 형태로 정리해줘"
"기능정의서 양식에 맞게 표로 만들어줘"
"노션 페이지로 옮겨줘"
```

AI한테 이렇게 시키면 돼요.

### 변환 예시

**마크다운 → 노션**

마크다운은 노션에 그대로 복붙하면 거의 그대로 들어가요.
표도 살아있고, 구조도 유지돼요.

**마크다운 → PPT**

AI한테 "이 내용으로 PPT 슬라이드 구성해줘" 하면
슬라이드별로 정리해줘요.

**마크다운 → 기능정의서 양식**

회사에서 쓰는 기능정의서 양식이 있으면,
그 양식에 맞게 재배치하라고 시키면 돼요.

근데 더 좋은 방법이 있어요.

**양식을 HTML로 먼저 만들어두세요.**

AI한테 "이 HTML 양식에 맞게 채워줘" 하면
깔끔하게 나와요. html -> pdf, ppt 로 해서 제출하면 됩니다. 

### 핵심: 순서가 중요하다

*(강조)*

여기서 핵심은 **순서**예요.

❌ 잘못된 순서:
```
처음부터 PPT로 만든다
→ 수정할 때마다 PPT 수정
→ 개발자한테 줄 때 또 정리
→ AI한테 줄 때 또 변환
```

✅ 올바른 순서:
```
마크다운으로 먼저 만든다 (AI가 읽는 형태)
→ 수정도 마크다운에서
→ 개발자한테 줄 때 그대로
→ 보고용으로 PPT 변환
```

**소스는 마크다운, 보고용은 변환해서.**

이게 바이브 기획의 워크플로우예요.

### 왜 이 순서인가?

이유가 있어요.

1. **마크다운은 수정이 빨라요** - 텍스트니까
2. **AI가 바로 이해해요** - 개발자한테 줄 때 변환 필요 없음
3. **변환은 한 번만** - 보고할 때만 PPT로

PPT를 소스로 쓰면?

수정할 때마다 PPT 열고, 텍스트 박스 찾고, 고치고...
AI한테 줄 때 또 복사해서 정리하고...

비효율이에요.

## [Part 4] 나만의 방식 만들기 - 24:00 - 28:00

### 기획 접근법 플로우

제가 기획할 때 쓰는 플로우를 보여드릴게요.

```
[아이디어]
    ↓
[1] 서비스 컨셉 정리 (/appkit.new)
    ↓
[2] 기능 구체화 (/appkit.spec)
    ↓
[3] 고객 여정 설계 (/appkit.customer)
    ↓
[4] 세일즈 메시지 (/appkit.sales)
    ↓
[5] MVP 범위 정의 (/appkit.mvp)
    ↓
[개발 착수]
```

각 단계마다 **마크다운 문서가 하나씩 생겨요.**

### 루프백: 이게 핵심이에요

근데 중요한 건요,
**이게 일직선이 아니에요.**

```
[1] 서비스 컨셉
    ↓
    결과 확인 → 아쉬움 발견
    ↓
    ┌──────────────────┐
    │  루프백 사이클    │
    │                  │
    │  프롬프트 수정    │
    │      ↓          │
    │  다시 실행       │
    │      ↓          │
    │  결과 비교       │
    │      ↓          │
    │  지침에 반영     │
    │      ↓          │
    │  (반복)         │
    └──────────────────┘
    ↓
    만족 → 다음 단계로
```

**매 단계에서 이 루프백을 돌려요.**

- 1단계 결과가 별로? → 프롬프트 고쳐서 다시
- 괜찮아졌어? → 그 프롬프트를 지침에 저장
- 다음에 또 쓸 때? → 저장된 지침으로 바로 실행

이게 **점점 똑똑해지는 시스템**이에요.

### 예시: 제가 루프백 돌린 과정

**처음 시도:**

> "야놀자처럼 테니스 코트 예약하는 앱 기획해줘"

결과? 별로였어요. 너무 일반적이었어요.

**루프백 1회차:**

> "서비스 컨셉, 타겟 고객, 핵심 문제, 해결책 정리해줘.
> 마크다운 표로."

좀 나아졌어요.

**루프백 2회차:**

> "이런 형태로:
> | 항목 | 내용 |
> |------|------|
> | 서비스명 | ... |"

이제 원하는 형태가 나왔어요.

**루프백 완료 → 저장:**

잘 된 프롬프트를 저장해뒀어요.

**5단계: 명령어로 만든다**

Custom Command로 만들어서
`/appkit.new`만 치면 같은 결과가 나오게.

### 핵심 공식

```
일단 시킨다
    → 결과 본다
        → 프롬프트 고친다
            → 다시 시킨다
                → 반복
                    → 저장
```

이 사이클을 돌리면 여러분만의 명령어가 만들어져요.

### 오늘부터 할 수 있는 것

*(구체적인 행동)*

**이번 주: 기록하기**

오늘부터 이것만 해보세요:

```

→ "00 정리해줘"

결과가 어땠지?
→ 70% 만족 / 표 형태가 아쉬움

다시 어떻게 물어봤지?
→ "마크다운 표로 다시"
```

메모장에 적어두세요.

**다음 주: 패턴 찾기**

기록 보면 패턴이 보여요:
- "나 맨날 '표로 정리해줘' 붙이네"
- "항상 '예시 들어서' 요청하네"

**그 다음: 템플릿 만들기**

자주 쓰는 프롬프트를 저장해두세요.

## [Q&A] 예상 질문

**Q1. 마크다운 어렵지 않나요?**

A: 5분이면 배워요. 제목은 `#`, 목록은 `-`, 표는 `|`로 구분.
노션 쓰시면 이미 마크다운 쓰고 계신 거예요.

**Q2. 회사에서 마크다운 안 받아주면요?**

A: 그래서 변환이 필요한 거예요.
마크다운으로 작업하고, 보고할 때만 PPT나 노션으로 변환하세요.
소스는 마크다운으로 유지하는 게 핵심이에요.

**Q3. 어떤 AI 도구를 써야 하나요?**

A: 지금 쓰고 계신 거 그대로 쓰세요.
도구보다 "마크다운으로 산출물 만들기" 습관이 더 중요해요.

**Q4. AppKit 그대로 써도 되나요?**

A: 참고용으로만 쓰세요.
제 업무 방식이라 여러분 업무엔 안 맞을 거예요.
구조랑 방법만 참고하시고, 반드시 여러분 버전을 만드세요.
