### 슬라이드 2: Claude Code의 3가지 구성요소

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   Claude Code = 3가지 자동화 도구
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1️⃣ Commands (커맨드)
   반복 작업을 명령어로 만들기
   ┌─────────────────────────────┐
   │ /appkit.new   아이디어 분해  │
   │ /appkit.spec  기능 명세      │
   │ /appkit.design API/DB 설계   │
   │                             │
   │ → 내 기획 프레임워크를       │
   │   커맨드로 저장              │
   └─────────────────────────────┘

2️⃣ SubAgents (서브 에이전트)
   복잡한 작업을 AI에게 위임
   ┌─────────────────────────────┐
   │ "이 기능 명세 검증해줘"      │
   │ "중복된 기능 찾아줘"         │
   │ "백로그 정리해줘"            │
   │                             │
   │ → AI가 알아서 분석/실행      │
   └─────────────────────────────┘

3️⃣ Skills (스킬)
   전문 작업을 AI에게 학습
   ┌─────────────────────────────┐
   │ "PRD 작성 스킬"              │
   │ "API 설계 스킬"              │
   │ "데이터 모델링 스킬"         │
   │                             │
   │ → 전문가 수준으로 작업       │
   └─────────────────────────────┘

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

핵심:
내 기획 방식을 Commands/SubAgents/Skills로 정리
→ Claude가 초지능 비서처럼 작동
```

**스크립트**:
> Claude Code는 3가지 구성요소로 이루어져 있습니다.
>
> **Commands**: 반복 작업을 명령어로 만듭니다. "/appkit.new"처럼 제가 자주 쓰는 기획 프레임워크를 커맨드로 저장했습니다.
>
> **SubAgents**: 복잡한 작업을 AI에게 위임합니다. "이 기능 명세 검증해줘"라고 하면 AI가 알아서 분석하고 실행합니다.
>
> **Skills**: 전문 작업을 AI에게 학습시킵니다. PRD 작성, API 설계 같은 전문 작업을 전문가 수준으로 수행합니다.
>
> 핵심은 **내 기획 방식을 이 3가지로 정리**하면, Claude가 **초지능 비서**가 된다는 겁니다.
>
> 오늘은 이 중에서 **Commands (appkit)**를 중심으로 보여드리겠습니다.

```

   PM/PO의 현실적 고민
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1️⃣ 산출물 관리의 혼란
   PRD_v1, v2, final, final_final...
   → 어디까지 했더라?
   → 이 기능 기획서 어디 있지?

2️⃣ 프로젝트 히스토리 부재
   담당자 바뀌면 맥락 파악 2주
   → 왜 이렇게 기획했지?
   → Notion, Jira, Slack 흩어짐

3️⃣ 기획 속도 느림
   바이브코딩: MVP 1주 완성
   기획: 문서만 2-3주 걸림
   → 개발은 빨라졌는데 기획이 병목
   → 속도 = 시장 경쟁력

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

해결책: 파일 기반 작업 + AI 자동화
```

**스크립트**:
> PM/PO 여러분, 이런 경험 있으시죠?
>
> 산출물 버전 관리 엉망, 담당자 바뀌면 맥락 파악 2주, 기획만 2-3주 걸려서 개발 병목.
>
> 요즘 바이브코딩 쓰면 MVP를 1주 만에 만드는데, 기획서 쓰는 데 2-3주 걸립니다.
>
> **속도가 시장 경쟁력인 시대**, 이 문제를 해결하는 게 바이브기획입니다.

### 슬라이드 4: ChatGPT vs Claude Code

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   왜 ChatGPT가 아니라 Claude Code인가?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ChatGPT (웹):
┌─────────────────────────────────┐
│ 아이디어 설명 → 채팅 응답       │
│ 결과 복사 → 워드 붙여넣기        │
│ 다음 날 → 맥락 다시 설명         │
│                                 │
│ 🔴 맥락 리셋, 일회성 작업         │
└─────────────────────────────────┘

Claude Code (파일 기반):
┌─────────────────────────────────┐
│ CLAUDE.md에 규칙 작성 (1회)     │
│ /appkit.new [아이디어]          │
│   → specs/*.md 자동 생성         │
│ 다음 날 → 그대로 이어서 작업     │
│                                 │
│ 🟢 맥락 유지, 누적 작업           │
│ 🟢 파일 = AI의 장기 기억          │
└─────────────────────────────────┘

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

핵심: CLAUDE.md = 프로젝트 위키
    → AI가 시작할 때마다 자동으로 읽음
```

**스크립트**:
> ChatGPT는 훌륭한 대화 도구입니다. 하지만 PM 업무에는 한계가 있습니다.
>
> 매번 맥락을 새로 설명해야 하고, 결과를 복사해서 붙여넣어야 합니다.
>
> Claude Code는 **파일 기반**입니다.
>
> CLAUDE.md에 프로젝트 규칙을 적어두면, AI가 시작할 때마다 자동으로 읽습니다.
>
> 오늘 작업한 내용이 내일도 유지되고, 3달 후에도 유지됩니다.
>
> **파일 = AI의 장기 기억**. 이게 결정적 차이입니다.

## 사용자 여정 (8단계)
1. 홈 화면 진입
2. 검색 조건 입력 (지역/날짜/시간)
3. 검색 버튼 클릭

5. 테니스장 선택
6. 시간대 선택

8. 결제 페이지 이동

## 비즈니스 규칙 (4가지)
- 예약 시간 제약: 현재+1시간 이후만 가능
- 중복 예약 방지: 동일 시간대 1건만
- 취소 정책: 24시간 전 무료, 12시간 전 50%
- 쿠폰 적용: 1회 1개만, 최소 금액 조건

## 예외 상황 (4가지)
- 검색 결과 0건 → 추천 제안
- 재고 마감 → 대기 예약 제안
- 결제 실패 → 10분 홀드 후 취소
- 동시 예약 충돌 → 먼저 결제한 사용자 우선

## API 엔드포인트
POST /api/bookings
  Request: { venueId, userId, dateTime, duration, couponId? }
  Response 200: { bookingId, status, totalAmount, finalAmount }
  Response 409: { error: "ALREADY_BOOKED" }

## DB 스키마
CREATE TABLE bookings (
  id UUID PRIMARY KEY,
  user_id UUID REFERENCES users(id),
  venue_id UUID REFERENCES venues(id),
  date_time TIMESTAMP NOT NULL,
  status VARCHAR(20),
  total_amount INT,
  discount_amount INT,
  final_amount INT,
  CONSTRAINT unique_venue_datetime UNIQUE(venue_id, date_time)
);

CREATE INDEX idx_bookings_user ON bookings(user_id);
CREATE INDEX idx_bookings_venue_date ON bookings(venue_id, date_time);
```

   "어디까지 했더라?"
   → 중복 작업, 누락 발생

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

바이브기획의 해결책:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1️⃣ 모든 것을 문서화
   specs/001-user.md
   specs/003-booking.md
   designs/api-endpoints.md
   backlog/future-features.md

   → 모든 산출물이 한 곳에 (프로젝트 폴더)

2️⃣ 모든 것을 백로그화
   /appkit.new로 초기 기능 목록 생성
   /appkit.spec으로 하나씩 구체화
   완료된 것 / 진행 중 / 미래 아이디어 분리

   → 진행 상황 한눈에 파악

3️⃣ Git으로 히스토리 자동 관리
   git commit -m "003-booking 취소 정책 24h→48h 변경"

   → 언제, 누가, 무엇을, 왜 바꿨는지 자동 추적
   → git log로 전체 히스토리 조회
   → git blame으로 특정 줄 변경 이력 추적

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

결과:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

산출물 베이스 히스토리
→ Claude가 초지능 비서처럼 작동

User: "003-booking 언제 만들었지?"
Claude: [git log 검색] "2025-11-15에 생성"

User: "취소 정책 왜 48시간으로 바꿨지?"
Claude: [git log 검색] "2025-11-20, 고객 요청으로 변경"

User: "원래대로 24시간으로 돌려줘"
Claude: [파일 수정 + git commit]
       "24시간으로 복구 완료"
```

**스크립트**:
> 프로젝트 매니징을 잘하려면 **문서화와 백로그화**가 핵심입니다.
>
> 전통적 방식은 산출물이 흩어지고, 히스토리가 없고, 백로그 관리가 안 됩니다.
>
> 바이브기획은 3가지로 해결합니다.
>
> **첫째, 모든 것을 문서화**: md 파일로 한 곳에 모음
>
> **둘째, 모든 것을 백로그화**: appkit으로 체계적 관리
>
> **셋째, Git으로 히스토리 자동 관리**: 언제 누가 왜 바꿨는지 자동 추적
>
> 결과는? **산출물 베이스 히스토리**가 쌓이고, Claude가 **초지능 비서**처럼 작동합니다.
>
> "언제 만들었지?" → git log 검색
>
> "왜 바꿨지?" → git log 검색
>
> "원래대로 돌려줘" → 파일 수정 + git commit
>
> **이게 바이브기획의 진짜 파워**입니다.
