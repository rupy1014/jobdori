# 바이브 기획: AI 시대, 기획자는 어떻게 일하는가

## 서론: 어떤 PM의 월요일 아침 (3분)

여러분, 월요일 아침에 출근하면 뭐부터 하세요?

예전에는 이랬죠.
Jira 열어서 티켓 확인하고,
Slack 밀린 메시지 읽고,
지난주 진행 상황 정리해서 주간 리포트 쓰고...

이거 하는 데만 오전이 다 갔어요.

근데 요즘 실리콘밸리 PM들은 다르게 일합니다.

일요일 밤에 AI 에이전트가 자동으로 돌아요.
Jira에서 진행 상황 긁어오고,
제품 사용 데이터 분석하고,
목표 대비 진척도 계산해서,
**월요일 아침 메일함에 주간 리포트가 와 있어요.**

PM은 출근해서 그 리포트 보면서 바로 전략 회의 들어갑니다.

이게 지금 일어나고 있는 일이에요.

---

### 숫자로 보는 변화

McKinsey에 따르면,
GenAI가 **PM 생산성을 40% 향상**시켰다고 합니다.

구체적으로 뭐가 달라졌냐면요,

- 문서 작성 시간: **수 시간 → 수 분**
- 고객 피드백 분석: **며칠 → 몇 시간**
- 스프린트당 절약 시간: **최대 18시간**

근데 이건 단순히 "빨라졌다"가 아니에요.
**일하는 방식 자체**가 바뀐 겁니다.
요구 되고 있죠
---

### 오늘 이야기할 것

오늘 저는 이 변화를 **"바이브 기획"**이라는 프레임으로 정리해서 말씀드리려고 해요.

바이브 기획이 뭐냐?
**AI와 대화하면서 함께 기획을 발전시켜 나가는 방식**입니다.

혼자 PRD 완성해서 던지는 게 아니라,
AI한테 생각을 던지고, 피드백 받고, 다듬고, 검증하고.

오늘 발표는 이렇게 진행됩니다:

1. **실제 사례**: AI 시대 PM들이 어떻게 일하는지
2. **도구와 방법**: 어떤 도구로, 어떻게 하는지
3. **프레임워크**: 바로 적용할 수 있는 실전 가이드

---

## 본론 1: 실제로 PM들이 어떻게 일하고 있나 (10분)

### 사례 1: 마케팅 자동화를 직접 만든 PM

HackerNoon에 올라온 실제 사례입니다.

한 PM이 이런 상황이었어요:
- 마케팅팀이 매번 데이터 추출해서
- HubSpot에 복사-붙여넣기 하고
- 템플릿 선택해서 이메일 발송

반복 작업이 너무 많았어요.
예전 같았으면 개발팀에 요청 티켓 넣고 3개월 기다려야 했겠죠.

근데 이 PM은 Cursor를 열었어요.
AI한테 이렇게 말했습니다:

> "우리 데이터베이스에서 고객 정보 가져와서
> HubSpot API로 자동 발송하는 스크립트 만들어줘"

**며칠 만에 자동화 완료.**
개발팀 리소스 안 쓰고요.

이 PM이 뭐라고 했냐면:
> "작은 업무들을 직접 빠르게 처리하니까,
> 더 많은 시간을 **전략적 업무**에 쓸 수 있게 됐다."

---

### 사례 2: 주간 리포트 자동화

Productboard 블로그에 나온 사례예요.

기존에 주간 리포트 쓰려면:
1. Jira에서 진행 상황 확인
2. 제품 사용 데이터 뽑기
3. 목표 대비 비교 분석
4. 이상 징후 파악
5. 문서로 정리

**이거 하는 데 반나절 걸렸어요.**

지금은?
n8n이라는 자동화 도구로 워크플로우 만들어서,
일요일 밤에 자동 실행.
**월요일 아침에 완성된 리포트가 메일로 옴.**

PM은 그 시간에 뭘 하냐?
**전략 수립과 의사결정.**

---

### 사례 3: BMAD 프레임워크를 쓰는 PM

이건 좀 더 체계적인 사례예요.

한 시니어 PM이 Obsidian(노트 앱)에
**8개의 전문화된 AI 에이전트**를 세팅해뒀어요.

예를 들어:
- 전략 기획 에이전트
- 가격 정책 에이전트
- 콘텐츠 작성 에이전트
- 경쟁사 분석 에이전트

헬스케어 데이터 제품 가격 정책을 고민할 때,
이렇게 물어봐요:

> "기업 고객 대상으로 가격 어떻게 책정해야 할까?"

에이전트가 구조화된 분석을 돌려줍니다.
이전에 했던 가격 정책 결정들도 참고해서요.

이 PM이 뭐라고 했냐면:
> "빈 문서 앞에서 멍하니 있는 시간이 사라졌다.
> **AI가 생각의 출발점**을 제공해준다."

---

### 사례 4: PM 면접이 바뀌고 있다

이건 좀 무서운 이야기인데요,

Google, Stripe, Netflix 같은 회사에서
**PM 채용 면접에 AI 프로토타이핑 테스트**가 들어가고 있어요.

면접관: "이 기능 아이디어를 프로토타입으로 만들어보세요."
시간: 30분

Lovable, Bolt, Cursor 같은 도구 쓸 수 있게 해주고,
**실제로 만들어보라고** 합니다.

왜 이런 테스트를 할까요?

> "PM이 제품 사고에서 **실행 가능한 프로토타입**으로
> 얼마나 빠르게 갈 수 있는지 보려는 것."

기획서만 잘 쓰는 PM은 더 이상 안 뽑겠다는 거예요.

---

### 무엇이 달라졌나: Before vs After

정리해볼게요.

| 업무 | Before | After |
|------|--------|-------|
| 주간 리포트 | 반나절 수작업 | 자동화, 검토만 |
| 피드백 분석 | 며칠 → 패턴 파악 | 몇 시간 → AI가 클러스터링 |
| PRD 초안 | 1-2주 작성 | 30분 대화 → 구조화된 초안 |
| 아이디어 검증 | 개발팀 대기 | 직접 프로토타입 |
| 경쟁사 분석 | 탭 수십 개 열고 정리 | AI가 수집 + 분석 |

핵심은 뭐냐면요,
**반복 작업은 AI가, PM은 판단과 전략에 집중.**

## 본론 2: 어떤 도구로 하는가 (7분)

### 기획 전용 AI의 한계

잠깐, 그러면 ChatGPT 쓰면 되는 거 아니야?
Notion AI 쓰면 되는 거 아니야?

되긴 됩니다. 근데 한계가 있어요.

**첫 번째 문제: 기획과 구현이 분리됨**
ChatGPT로 PRD 뽑아도,
개발할 때 다시 설명해야 해요.
기획 AI와 개발 AI가 따로 놀아요.

**두 번째 문제: 컨텍스트가 날아감**
오늘 작업한 거, 다음 주에 이어서 하려면?
처음부터 다시 설명해야 해요.

**세 번째 문제: 검증이 안 됨**
"이거 기술적으로 가능해?" → 기획 AI는 모릅니다.
실제 코드를 모르니까요.

---

### 앞서가는 PM들이 쓰는 도구

그래서 실제로 PM들이 쓰는 조합을 보면요,

**1단계: 아이디어 + 대화**
- ChatGPT, Claude
- 생각 정리, 브레인스토밍, 초안 작성

**2단계: 프로토타이핑 + 구현**
- Cursor, Claude Code
- 실제 코드 생성, 자동화 구축

**3단계: 문서화 + 컨텍스트 유지**
- CLAUDE.md, Obsidian
- 프로젝트 맥락을 파일로 저장

핵심이 뭐냐면요,
**기획-검증-구현이 하나의 맥락에서 이어지는 것.**

---

### Claude Code / Cursor의 차별점

제가 직접 써보면서 느낀 건데요,

**첫째, 기획하면서 바로 검증됩니다.**

"이 기능 복잡해?" → 바로 답변
"이거 빼면 얼마나 빨라져?" → 바로 계산
"이 구조로 하면 나중에 문제 없어?" → 바로 확인

**둘째, 맥락이 유지됩니다.**

CLAUDE.md 파일에 프로젝트 정보를 적어두면,
한 달 후에 열어도 AI가 기억합니다.
신규 입사자가 AI에게 물어보면 프로젝트 설명도 해줘요.

**셋째, 우리 팀에 맞게 커스터마이징 가능합니다.**

우리 회사 기획 템플릿,
우리 팀의 규칙,
이런 것들을 AI에게 학습시킬 수 있어요.

## 본론 3: 어떻게 하는가 - 바이브 기획 프레임워크 (8분)

### 바이브 기획의 핵심 원칙

실제 PM들 사례를 보면서 패턴이 보였어요.
세 가지로 정리해봤습니다.

**원칙 1: 던지고 다듬어라**

기존: 완벽한 문서를 쓰려고 함
바이브: 생각을 그냥 던지고, AI랑 같이 발전시킴

예시:
```
"음... 결제 이탈률이 높은데,
간편하게 만들 방법 없을까?"
```

이렇게 던지면 AI가 구조화해줍니다.
거기서 대화하면서 다듬어가는 거예요.

**원칙 2: 비판을 요청해라**

AI한테 칭찬 받으려고 하지 마세요.
오히려 이렇게 물어보세요:

- "이 기획에서 빠진 게 뭐야?"
- "사용자 입장에서 짜증날 포인트가 뭐야?"
- "이거 개발하면 얼마나 걸려?"

혼자서는 못 보는 구멍을 AI가 찾아줍니다.

**원칙 3: 작게 쪼개라**

AI는 큰 거 한 번에 못 합니다.
작은 단위로 쪼개서 시켜야 해요.

```
❌ "회원 시스템 만들어줘"

✅ "회원가입에 필요한 필드가 뭐가 있을까?"
✅ "이메일 인증 플로우 그려줘"
✅ "비밀번호 정책은 어떻게 할까?"
```

---

### 실전 가이드: 바이브 기획 4단계

#### 1단계: 컨셉 대화 (30분)

목표: 아이디어를 구체화하기

AI한테 이렇게 시작해요:
> "나 지금 [이런 기능] 고민 중인데,
> 같이 정리 좀 해줘."

그러면 AI가 질문을 던져요:
- 해결하려는 문제가 뭔가요?
- 타겟 사용자가 누구예요?
- 기존 방식과 뭐가 다른가요?

대화 끝나면 **concept.md** 파일로 정리.

#### 2단계: 스펙 구체화 (1-2시간)

목표: AI도 이해하는 기획서 만들기

이 단계에서 **프로젝트 바이블**을 만듭니다.
Claude Code에서는 CLAUDE.md라고 부르는데요,

들어가는 내용:
- 프로젝트 목적과 배경
- 핵심 기능 목록
- 사용자 시나리오
- **하면 안 되는 것들** (이게 중요!)

왜 "하면 안 되는 것"이 중요하냐면요,

```
❌ "간단하게 만들어줘" → 해석이 다양함
✅ "OAuth 외에 다른 로그인 방식 추가하지 마" → 명확함
```

AI는 하라는 것보다 **하지 말라는 것**을 더 잘 따릅니다.

#### 3단계: 검증 (30분-1시간)

목표: 기획의 허점 찾기

AI한테 비판을 요청해요:
- "이 기획에서 빠진 거 뭐야?"
- "엣지 케이스 뭐가 있어?"
- "개발 난이도 어때?"

그리고 실제로 간단한 프로토타입을 만들어봐요.
Cursor나 Claude Code로 30분이면
기본 화면 정도는 볼 수 있어요.

**만들어보면 기획의 허점이 바로 보입니다.**

#### 4단계: 태스크 분해 (30분)

목표: 실행 가능한 단위로 쪼개기

큰 기능을 작은 태스크로 나눠요.
각 태스크는 "한 번의 대화에서 끝낼 수 있는 크기"여야 해요.

예시:
```
[회원 관리 시스템]
ㄴ 1. 회원가입 화면 기획
ㄴ 2. 이메일 인증 플로우 정의
ㄴ 3. 로그인/로그아웃 흐름
ㄴ 4. 비밀번호 찾기 프로세스
ㄴ 5. 회원 정보 수정 화면
```

이렇게 쪼개면 AI도 잘 따라오고,
진행 상황 추적도 쉬워요.

## 결론: 오늘부터 시작하기 (2분)

### 핵심 정리

**1. PM의 일하는 방식이 바뀌고 있다**
- 반복 작업 → AI가 처리
- PM → 판단, 전략, 방향 설정에 집중
- 주간 리포트부터 프로토타이핑까지 자동화

**2. 도구가 중요하다**
- 기획 AI 따로, 개발 AI 따로 ❌
- 기획-검증-구현이 연결된 도구 ✅
- 맥락이 유지되는 환경 구축

**3. 바이브 기획 = 대화하며 함께 만들기**
- 완벽한 문서 ❌ → 던지고 다듬기 ✅
- 혼자 고민 ❌ → AI한테 비판 요청 ✅
- 큰 덩어리 ❌ → 작게 쪼개기 ✅

### 오늘 퇴근 후 해볼 것

지금 진행 중인 프로젝트 하나 떠올려보세요.

Claude나 ChatGPT 열고,
이렇게 말해보세요:

> "나 지금 [프로젝트] 기획하고 있는데,
> 내가 놓치고 있는 게 뭐가 있을까?"

이게 바이브 기획의 시작입니다.

### 마무리

AI가 코드를 짜는 시대,
기획자는 뭘 해야 할까요?

**"무엇을 만들지"를 정하는 것.**
**"왜 만드는지"를 설명하는 것.**
**"어떻게 만들지"를 AI와 함께 설계하는 것.**

그리고 바이브 기획은,
이 세 가지를 **더 빠르고, 더 정확하게** 하는 방법입니다.

감사합니다.
